# QA Assessment App — dev workflow

Question/answer assessment app. Admins manage multiple-choice questions
(single- or multi-select); users take an assessment one question at a time
and get a score at the end, computed server-side.

Repo: https://github.com/Ali-zain-javed/qa-assessment-app (main branch)

## Layout

```
backend/           FastAPI + SQLAlchemy + Postgres
  app/core/         config.py (env settings), security.py (JWT/bcrypt),
                     database.py (SQLAlchemy engine/session), deps.py (auth guards)
  app/models/        User, Question, Attempt (SQLAlchemy models)
  app/schemas/       Pydantic request/response schemas, 1:1 with models/
  app/routers/       auth.py, questions.py, attempts.py
  app/main.py        FastAPI app, CORS, router registration
  app/seed.py        creates tables + demo admin/user/questions
  .env               DATABASE_URL, SECRET_KEY (gitignored — see .env.example)

frontend/           React + Vite + Tailwind v4
  src/api.js          fetch wrapper for every backend call
  src/AuthContext.jsx  JWT auth state (login/logout/current user), localStorage-backed
  src/App.jsx          routes + role-based route guards (admin vs user)
  src/pages/           Login.jsx, AdminDashboard.jsx, Assessment.jsx
  src/components/      QuestionCard.jsx (single question, single/multi select UI)
  .env                 VITE_API_URL (gitignored — see .env.example)
```

## Running it

Backend (needs Postgres running — `brew services start postgresql@18` if not):

```bash
cd backend
source .venv/bin/activate      # venv was built with Python 3.13, not 3.14 — see Gotchas
uvicorn app.main:app --reload --port 8000
```

Frontend:

```bash
cd frontend
npm run dev                     # http://localhost:5173
```

Reset/reseed the DB (creates tables if missing, adds demo admin/user/questions,
safe to re-run — it no-ops if that admin/user/question data already exists):

```bash
cd backend && source .venv/bin/activate && python -m app.seed
```

## How auth/scoring work (read before changing either)

- One login endpoint (`POST /auth/login`) for everyone. JWT payload carries
  `sub` (user id); role (`admin`/`user`) lives on the `users` row and is
  returned alongside the token, not encoded in the JWT itself.
- `GET /questions` (user-facing) never returns `correct_options` — see
  `QuestionOut` vs `QuestionAdminOut` in `schemas/question.py`. Only
  `GET /admin/questions` includes them. Do not add correct answers to the
  user-facing schema.
- Scoring happens entirely in `POST /attempts/submit`
  (`routers/attempts.py`) by comparing submitted `selected_options` against
  each question's stored `correct_options` — the frontend just posts
  `{question_id, selected_options}` pairs and displays whatever score comes
  back. Don't compute scores client-side.

## Common changes

- **New question field or type**: update `models/question.py`,
  `schemas/question.py`, then `AdminDashboard.jsx`'s form and
  `QuestionCard.jsx`'s render. No Alembic migration exists yet (see Gotchas)
  — drop/recreate the table or hand-write the ALTER for existing data.
- **New protected endpoint**: add to the right router, depend on
  `get_current_user` or `require_admin` from `core/deps.py`.
- **New frontend page**: add to `src/pages/`, wire into `App.jsx`'s
  `<Routes>`, wrap in `<ProtectedRoute role="...">` if it needs auth.

## Gotchas

- **Python version**: the backend venv must be built with Python 3.13
  (`/opt/homebrew/bin/python3.13 -m venv .venv`). Python 3.14 fails to build
  `pydantic-core`/`psycopg2-binary` wheels as of this writing.
- **No Alembic yet**: schema is created via `Base.metadata.create_all()` in
  `seed.py`, not migrations. Fine for dev; add Alembic before any real
  deployment so schema changes don't require dropping tables.
- **DB credentials**: `backend/.env` currently points at
  `postgresql://fastapi_user:password123@localhost:5432/fastapi_db`. That
  role needed `GRANT ALL ON SCHEMA public` run once locally — if you spin up
  a fresh Postgres instance/role, re-run that grant or you'll hit
  `permission denied for schema public` on `python -m app.seed`.
- **Tailwind v4**: uses `@tailwindcss/postcss` in `postcss.config.js` and
  `@import "tailwindcss";` in `index.css` — not the v3
  `tailwind.config.js` + `@tailwind base/components/utilities` pattern.

## Deployment

Not yet deployed. Target is AWS Elastic Beanstalk (discussed, not started).
Because this is two separate apps (FastAPI API + static React build) under
one EB app, expect to either: (a) deploy backend to EB and frontend to
S3/CloudFront, or (b) containerize both and run as multi-container EB.
Decide this with the user before scaffolding — don't assume.
